# Standard Operating Procedures (SOP)

**Project:** inteteam_cms — Multi-Tenant Website Builder for UK Repair Shops
**Company:** Inte.Team
**Target Scale:** 500+ shops Phase 1 → Wix-style platform long-term
**Last Updated:** 2026-03-20
**Version:** 1.0

---

## Purpose

This SOP provides clear, actionable procedures for:
1. **Feature Planning** — 9-step process to plan and document features correctly the first time
2. **Granular Debugging** — Step-by-step methodology to isolate and fix issues
3. **Architectural Analysis** — How to evaluate new features against scalability constraints

**Rule:** Read this file at the start of every planning session, debugging session, or architectural discussion. Do not rely on memory of previous sessions.

### Mandatory Patterns (never deviate)

| Concern | Pattern |
|---------|---------|
| All PHP files | `declare(strict_types=1);` |
| Primary keys | `HasUlids` trait |
| Tenant scoping | `HasCompanyScope` trait on every tenant model |
| Authorization | Policy class + `abort(403)` in controller — NEVER inline `if` checks |
| Validation | Form Request classes, array syntax rules — NEVER pipe strings |
| Test framework | PHPUnit — `final class XTest extends TestCase` — NEVER Pest |
| File uploads | `Route::match(['PUT','POST'],...)` + `post()` on frontend |

| Services | `final` class, constructor injection, <250 lines |
| Controllers | Thin orchestrators only, <150 lines |
| Component size | Atoms <20 lines, Molecules <50, Organisms <150 |

## Feature Planning Procedure

### Step 0 — Read Before Writing (MANDATORY)

Before writing a single line of documentation or code:

1. Re-read `docs/planning/README.md` (architecture decisions, DB schema, block types, phase plan)
2. Re-read the relevant `docs/features/{name}/README.md` if updating an existing feature
3. Re-read this SOP file

Verify you can answer:
- [ ] What is the tenant isolation pattern? (`HasCompanyScope` — auto `company_id` scoping)
- [ ] What is the exact flash message format? (`->with(['alert' => '...', 'type' => 'success'])`)
- [ ] Where do blocks live? (JSON column on `cms_pages`/`cms_posts` — no separate table)
- [ ] What is the auth guard name? (`cms` — not `web`)
- [ ] What test framework? (PHPUnit, `final class extends TestCase`, `use RefreshDatabase`)
- [ ] How do CRM blocks render? (Server-side via `BlockRendererService` + `CrmApiClient` + Redis cache)
- [ ] How are new block types registered? (Via `BlockTypeRegistry::register()` at boot — NOT by editing an enum)

**Stop point:** If you cannot answer these, read the docs again before proceeding.

### Step 1 — Define Business Requirements

**Output:** Feature README in `docs/features/{name}/README.md`

1. Create feature directory: `mkdir -p docs/features/{name}`
2. Answer these questions in the README:
   - [ ] What problem does this solve for the shop owner?
   - [ ] User stories with acceptance criteria
   - [ ] What block types or CRM integration does it touch?
   - [ ] Does it affect public site rendering (SEO implications)?
   - [ ] Does it need to work for anonymous visitors (public) or only admins?

---

### Step 2 — Design Technical Architecture

**Output:** Architecture section in the feature README

1. Identify all affected layers:
   - [ ] Database tables / columns (new or modified)
   - [ ] Models (new or modified)
   - [ ] Services
   - [ ] DTOs
   - [ ] Controllers + routes
   - [ ] Policies
   - [ ] Frontend pages (Inertia)
   - [ ] Blade partials (public site)

2. Check architectural constraints:
   - [ ] Multi-tenancy: every new table needs `company_id` + `HasCompanyScope`
   - [ ] Indexes: every table needs `idx_{table}_company_created (company_id, created_at)` at minimum
   - [ ] Caching: does this affect public page rendering? If yes, plan cache key + TTL + bust event
   - [ ] Block types: if adding a new block type, does it go through `BlockTypeRegistry`?
   - [ ] CRM blocks: if calling CRM API, is the response cached? Is the error state handled gracefully?
   - [ ] SEO: does this affect `<head>` output? Does it need a JSON-LD schema type?
   - [ ] Queue: any operation >2s must be queued

3. Scalability checkpoint (see Architectural Analysis section below)

---

### Step 3 — Plan Database Changes

**Output:** Schema documented in the feature README

1. For every new table, document:
   - All columns with types and constraints
   - Mandatory indexes:
     - `idx_{table}_company_created (company_id, created_at)`
     - `idx_{table}_company_status (company_id, status)` if status column exists
     - `idx_{table}_company_{slug} (company_id, slug)` if slug column exists
   - Foreign key `CASCADE` rules (delete cascade vs. set null)
   - `HasCompanyScope` if tenant-scoped
   - `HasUlids` for primary key

2. Foreign key type rule: if the parent PK is a ULID → `foreignUlid()`. If bigint → `foreignId()`. Check the parent model's traits.

3. Never add a JSON catch-all column where explicit columns will do. Only use JSON for genuinely variable-schema data (blocks, opening hours, nav items).

### Step 4 — Inventory Components Before Planning

**MANDATORY: List existing components before planning any new ones.**

```bash
# When code exists, list before planning:
ls resources/js/Components/
ls resources/js/Pages/Admin/
ls resources/views/blocks/
```

DRY rules:
- ❌ Never plan a Button if `ui/button.tsx` exists
- ❌ Never plan a new block Blade partial without checking `resources/views/blocks/`
- ✅ Document which existing components are reused and their import paths
- ✅ Only plan truly new components

Component structure:
```
resources/js/
├── Components/
│   ├── Atoms/        <20 lines each
│   ├── Molecules/    <50 lines each
│   ├── Organisms/    <150 lines each
│   ├── BlockEditor/  Block editor components
│   └── ui/           Base shadcn/radix components
├── Pages/Admin/      Inertia admin pages
resources/views/
├── layouts/          Blade layouts
├── partials/         Shared Blade partials
└── blocks/           One partial per block type
```

### Step 5 — Plan Block Types (if applicable)

If the feature involves new block types:

1. Define the block's JSON `data` shape (document in `features/blocks/README.md`)
2. Register in `BlockTypeRegistry` (not an enum — see Architecture section below)
3. Write the Blade partial: `resources/views/blocks/{type}.blade.php`
4. Write the error partial: `resources/views/blocks/{type}_error.blade.php`
5. Write the React editor component: `resources/js/Components/BlockEditor/blocks/{Type}Block.tsx`
6. If CRM block: add method to `CrmApiClient`, define cache key + TTL
7. Validate `data` shape in `BlockDataValidator`

### Step 6 — Write Tests First (TDD)

**Output:** Failing test files in `tests/Feature/{Area}/`

Test every:
- [ ] Authentication / Authorization (guest → 401, wrong role → 403, correct role → 200)
- [ ] CRUD operations (valid input, invalid input, boundary values)
- [ ] Multi-tenant isolation: company A cannot access company B's records
- [ ] Business logic rules (slug uniqueness, type uniqueness, publish rules)
- [ ] Public site: drafts return 404, published content returns 200
- [ ] Cache: published content is cached, bust fires on update/delete
- [ ] Block rendering: CRM blocks show error partial on API failure (don't break the page)

Run to confirm tests fail before implementing:
```bash
docker compose exec php-fpm php artisan test --filter={Feature}Test
```

### Step 7 — Implement

Implementation order (always this order — dependencies flow downward):

```
1. Migration
2. Model (HasUlids, HasCompanyScope, casts, scopes, relationships)
3. Factory
4. DTO(s)
5. Form Request(s)
6. Policy
7. Service
8. Controller
9. Routes
10. Inertia pages (admin UI)
11. Blade partials (public site, if applicable)
```

TDD cycle: write failing test → minimal code to pass → refactor → repeat.

## Granular Debugging Procedure

### Rules

- **Never run the full test suite at once.** Run by feature or by directory.
- **Surgical fixes only.** Do not rewrite architecture to fix a bug.
- **90% of integration bugs are field name mismatches** — check migration docs before assuming logic is wrong.

### Level 1 — Reproduce

1. What exact action triggers it?
2. Is it consistent or intermittent?
3. Check logs immediately:
   ```bash
   docker compose logs php-fpm --tail=50
   tail -f storage/logs/laravel-$(date +%Y-%m-%d).log
   ```

### Level 2 — Identify the Layer

| Symptom | Likely layer |
|---------|-------------|
| 500 with stack trace | Service or Model |
| 422 Unprocessable Entity | Form Request validation |
| 403 Forbidden | Policy |
| 404 Not Found | Route or `HasCompanyScope` filtering out the record |
| Empty data, no error | `HasCompanyScope` with wrong `company_id` in session |
| Block renders blank | `BlockRendererService` swallowing an exception silently |
| CRM block shows error partial | `CrmApiClient` timeout or bad API key — check `crm_api_key` on company |

### Level 3 — Isolate

Add targeted logging at each layer boundary:

```php
Log::debug('PageController@show', ['slug' => $slug, 'company_id' => $company->id]);
Log::debug('PageService::findBySlug result', ['found' => $page !== null]);
```

Use Tinker to inspect state:
```bash
docker compose exec php-fpm php artisan tinker
```

```php
// Bypass company scope to confirm record exists
CmsPage::withoutGlobalScope(CompanyScope::class)->where('slug', 'about')->first();

// Check cache
Cache::get("cms:page:{$companyId}:about");

// Test CRM client directly
$client = app(CrmApiClientFactory::class)->forCompany($company);
$client->testConnection();
```

### Level 4 — Check Database State

```sql
-- Confirm company_id is set correctly
SELECT id, slug, company_id, status FROM cms_pages WHERE slug = 'about';

-- Check for orphaned records
SELECT p.* FROM cms_pages p
LEFT JOIN companies c ON p.company_id = c.id
WHERE c.id IS NULL;

-- Check actual indexes
SHOW INDEXES FROM cms_pages;
```

### Level 5 — Write a Reproducing Test

Write the smallest failing test that proves the bug:

```php
public function test_published_page_is_accessible_at_slug(): void
{
    $company = Company::factory()->create(['slug' => 'acme']);
    $page = CmsPage::factory()->for($company)->published()->create(['slug' => 'about']);

    $response = $this->get('http://acme.cms.inte.team/about');

    $response->assertOk();
}
```

Fix the bug, verify the test passes, verify no regressions in related tests.

## Architectural Analysis Procedure

Use this when evaluating whether a proposed feature fits the architecture, or when assessing a potential future direction (e.g. scaling to Wix-style).

### Checklist

**Multi-tenancy:**
- [ ] Is every new table scoped by `company_id`?
- [ ] Does every query path go through `HasCompanyScope`?
- [ ] Is there any route where company context is ambiguous?

**Block system extensibility:**
- [ ] Is the new block type registered via `BlockTypeRegistry::register()` — not hardcoded?
- [ ] Is the block's data shape validated in `BlockDataValidator`?
- [ ] Is the Blade partial isolated (no global state, no DB calls — data passed in)?

**SEO:**
- [ ] Does every new public URL have a canonical, title, and description resolved by `SeoMetaService`?
- [ ] Is the new content type included in the sitemap controller?
- [ ] If new content type has structured data potential, is a JSON-LD schema planned?

**Performance:**
- [ ] Is every public-facing read cached? What is the TTL? What busts the cache?
- [ ] Are N+1 queries possible on any list view? Use `with()` and measure.
- [ ] Does any synchronous operation risk >2s? If yes, move to queued job.

**CRM integration:**
- [ ] Is the CRM API response cached?
- [ ] Is there a graceful error state if the CRM is unreachable?
- [ ] Is the API key never exposed in HTML or JavaScript?

**Wix-scale readiness (see section below):**
- [ ] Is this feature gated by the block registry (not an enum)?
- [ ] Is the public site theme referenced by name from `companies.theme` (not hardcoded)?
- [ ] Is any new company-level capability gated by `companies.plan` or SSO subscription tier?

**Revisions & staging:**
- [ ] Does any write operation that touches `blocks` go through `RevisionService::stagePageRevision()`?
- [ ] Is the live `cms_pages.blocks` column only ever written to by `RevisionService::publishStaged()`?
- [ ] Do MCP write tools call `RevisionService` — never write directly to the page/post model?

**MCP tools:**
- [ ] Does the new tool implement the `McpTool` interface?
- [ ] Is it registered in `McpToolRegistry` with the correct permission level?
- [ ] Does it return structured error responses (not exceptions)?
- [ ] If it's a write tool, does the description explicitly state it stages and not publish?

## Wix-Scale Architecture Notes

These are the three patterns to enforce from Phase 1 to avoid rewrites later:

### 1. Block Type Registry (not an enum)

**Wrong (hardcoded enum — breaks at scale):**
```php
enum BlockType: string
{
    case RichText = 'rich_text';
    case Image = 'image';
    // Every new block type requires editing this file
}
```

**Right (registry — new types register themselves):**
```php
final class BlockTypeRegistry
{
    private static array $types = [];

    public static function register(string $type, string $rendererClass): void
    {
        static::$types[$type] = $rendererClass;
    }

    public static function all(): array
    {
        return array_keys(static::$types);
    }

    public static function renderer(string $type): ?string
    {
        return static::$types[$type] ?? null;
    }
}
```

Block types register in a `AppServiceProvider::boot()` call. Adding a new block type never requires touching existing files. This is the foundation for a future block marketplace.

### 2. Themes as a First-Class Concept

**Wrong (hardcoded):**
```php
// Blade layout always loads the same templates
// resources/views/layouts/public.blade.php — hardcoded design
```

**Right (theme name on company, resolved at render time):**
```php
// companies.theme = 'default' | 'minimal' | 'bold' (future)
$theme = $company->theme ?? 'default';
view()->share('theme', $theme);

// Blade templates use theme-aware paths:
// resources/views/themes/{theme}/layouts/public.blade.php
// resources/views/themes/{theme}/blocks/rich_text.blade.php
```

In Phase 1 only the `default` theme exists. The directory structure is already correct. Adding a second theme is copying the directory and editing CSS — no PHP changes.

### 3. Plan Column on Companies

```sql
-- Add from day 1
plan  ENUM('starter','standard','pro','enterprise') DEFAULT 'starter'
```

Used to gate features per company tier. Even if you don't have tiers in Phase 1, every feature gating check should read `$company->plan` rather than hardcoded `true`. When SSO is integrated, this syncs from the SSO subscription claim. When you have 10,000 tenants with different capabilities, this column is what you query.

---

### What Would Actually Need to Change for Wix-Scale

Current architecture handles these without changes:
- More companies (multi-tenancy scales linearly)
- More block types (BlockTypeRegistry)
- More themes (theme directory pattern)
- Headless API for Next.js (Phase 4, already planned)
- SSO (Phase 3, already planned)

These would need new features (not rewrites):
- **Member-gated pages** — `cms_members` table + `is_members_only` on pages
- **App/plugin marketplace** — `app_installations` table, block types installed per company
- **Analytics** — `page_views` time-series table (or delegate to external like Plausible)
- **Native form builder** — `cms_forms` + `cms_form_fields` tables (not CRM-borrowed)
- **Multi-language** — `cms_page_translations` pivot table

Nothing in Phase 1 prevents these. They are additive, not corrective.
